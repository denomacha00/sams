# Requirements Document

## Introduction

SAMS (Smart Attendance Management System) is a multi-school, enterprise-grade attendance platform for educational institutions. The system prevents fraudulent attendance through multi-layer validation (QR code, GPS, and biometric), tracks real-time attendance per session, and supports multiple schools under a single platform with complete data isolation.

The platform operates across three subdomains:
- `smart-managment.com` — Main React application and public website
- `api.smart-managment.com` — Express.js backend API
- `super.smart-managment.com` — Private Super Admin control panel

SAMS is designed for cloud deployment on a VPS (Ubuntu 22.04), supports offline-first operation with automatic sync, integrates role-aware AI assistance using free browser-based tools, and scales to biometric and mobile integrations. Payments are confirmed manually by the Super Admin. All environment variables are pre-baked into the deployment configuration.

## Glossary

- **SAMS**: Smart Attendance Management System — the platform described in this document.
- **School**: A registered educational institution activated via a license key.
- **License_Key**: A one-time-use activation token in the format `XXXX-YYYY-XXXX-XXXX` (e.g., `MTTI-2026-X9K2-4B7A`) that encodes school name, plan tier, and expiry date.
- **School_Code**: A short, memorable identifier chosen by the School Admin during activation (e.g., `mukiria`), used by all school users to log in.
- **Super_Admin**: The developer-level administrator who manages all schools, generates licenses, and has exclusive access to `super.smart-managment.com`.
- **School_Admin**: The top-level administrator within a single school, responsible for managing HODs, teachers, students, timetables, and school-wide reports.
- **HOD**: Head of Department — manages teachers and students within a specific department.
- **Teacher**: Manages students in an assigned class, starts attendance sessions, and marks attendance.
- **Student**: End-user who scans QR codes and views personal attendance records.
- **Attendance_Session**: A time-bounded event started by a Teacher for a specific class and subject, during which attendance is recorded.
- **Registration_Link**: A URL generated by a role holder that embeds School_ID, Department_ID, Class_ID, and expiry metadata, used for self-registration.
- **QR_Code**: A dynamically refreshing code displayed by the Teacher's device, scanned by students to mark attendance.
- **Biometric_Template**: A stored facial or fingerprint data record associated with a Student, used for biometric attendance verification.
- **Liveness_Detection**: A biometric check that confirms the subject is a live person, not a photo or recording.
- **Dropout_Risk_Score**: A computed value (0–100%) indicating a student's risk of dropping out, derived from attendance, grades, and behavioral patterns.
- **IndexedDB**: Browser-based local storage used for offline data persistence on student and teacher devices.
- **Conflict_Resolution**: The process of reconciling attendance records created both offline and online.
- **Plan_Tier**: One of four subscription levels: Trial, Basic, Professional, or Enterprise.
- **Late_Threshold**: A configurable time window (in minutes) after session start within which a student is marked PRESENT rather than LATE.
- **Attendance_Status**: One of four values: PRESENT, LATE, EXCUSED, or ABSENT.
- **AI_Assistant**: The role-aware assistant embedded in SAMS that answers queries based on the user's permission level using free, browser-based tools.
- **Timetable**: A school-wide schedule mapping classes, subjects, teachers, and time slots.
- **AuditLog**: An immutable record of significant system events for compliance and debugging.
- **FaceAPI**: FaceAPI.js — a browser-based facial recognition library used for biometric attendance.
- **Web_Speech_API**: A browser-native API used for voice input to the AI Assistant.
- **VPS**: Virtual Private Server at IP `185.143.228.182` running Ubuntu 22.04.
- **JWT**: JSON Web Token used for stateless authentication.
- **Socket_IO**: WebSocket library used for real-time attendance feed updates.

### Requirement 1: License Key Generation and Activation

**User Story:** As a School Admin, I want to activate my school using a license key generated by the Super Admin, so that my institution is registered on SAMS with the correct plan tier and data isolation.

#### Acceptance Criteria

1. WHEN the Super Admin generates a license key, THE License_Generator SHALL encode the school name, plan tier, and expiry date into a key matching the format `XXXX-YYYY-XXXX-XXXX` and SHALL store a hashed reference in the database.
2. WHEN a user submits a valid, unused, non-expired license key at the activation endpoint, THE Activation_Service SHALL register the school, create the School Admin account, and mark the license key as used.
3. WHEN a license key is submitted that has already been used, THE Activation_Service SHALL reject the request and return a descriptive error message.
4. WHEN a license key is submitted that is expired, THE Activation_Service SHALL reject the request and return a descriptive error message.
5. WHEN a license key is submitted that does not match the format `XXXX-YYYY-XXXX-XXXX`, THE Activation_Service SHALL reject the request and return a descriptive error message.
6. THE Activation_Service SHALL never expose the raw license key value in any API response, log entry, or UI element after activation.
7. WHEN activation succeeds, THE Activation_Service SHALL store the School_Code chosen by the School Admin and enforce its uniqueness across all schools.
8. WHEN a School_Code collision is detected during activation, THE Activation_Service SHALL reject the submission and prompt the School Admin to choose a different code.

### Requirement 2: Multi-School Data Isolation

**User Story:** As a Super Admin, I want each school's data to be fully isolated, so that users from one school cannot access or affect data belonging to another school.

#### Acceptance Criteria

1. THE Data_Layer SHALL associate every School, User, Department, Class, Timetable, Attendance_Session, AttendanceRecord, Registration_Link, Payment, and AuditLog record with a `schoolId` field.
2. WHEN any authenticated user makes a data request, THE API SHALL scope all query results to the `schoolId` embedded in the user's JWT token.
3. IF a request attempts to access a resource whose `schoolId` does not match the requesting user's `schoolId`, THEN THE API SHALL return a 403 Forbidden response.
4. THE Super_Admin_Panel SHALL be accessible only via `super.smart-managment.com` and SHALL require Super Admin credentials that are not associated with any single school.
5. WHEN the Super Admin views school data, THE Super_Admin_Panel SHALL display per-school data without merging records across schools.

### Requirement 3: Role-Based Access Control

**User Story:** As a School Admin, I want each user role to have clearly defined permissions, so that users can only perform actions appropriate to their role.

#### Acceptance Criteria

1. THE Auth_Service SHALL assign exactly one of the following roles to each user: Super_Admin, School_Admin, HOD, Teacher, or Student.
2. WHEN a School_Admin attempts to manage HODs, Teachers, or Students within the same school, THE API SHALL permit the operation.
3. WHEN a HOD attempts to manage Teachers or Students outside the HOD's assigned department, THE API SHALL return a 403 Forbidden response.
4. WHEN a Teacher attempts to start an Attendance_Session for a class not assigned to that Teacher, THE API SHALL return a 403 Forbidden response.
5. WHEN a Student attempts to access another student's attendance records, THE API SHALL return a 403 Forbidden response.
6. THE Auth_Service SHALL embed the user's role, schoolId, departmentId, and classId in the JWT token payload.
7. WHEN a JWT token expires, THE Auth_Service SHALL reject all API requests using that token and return a 401 Unauthorized response.
8. THE Auth_Service SHALL support token refresh using a valid refresh token without requiring the user to re-enter credentials.

### Requirement 4: Hierarchical Registration Links

**User Story:** As a Teacher, I want to generate a registration link for my class, so that students can self-register with their school, department, and class pre-filled.

#### Acceptance Criteria

1. WHEN a School_Admin generates a registration link, THE Registration_Service SHALL embed the schoolId and set the target role to HOD.
2. WHEN a HOD generates a registration link, THE Registration_Service SHALL embed the schoolId and departmentId and set the target role to Teacher.
3. WHEN a Teacher generates a registration link, THE Registration_Service SHALL embed the schoolId, departmentId, and classId and set the target role to Student.
4. THE Registration_Service SHALL enforce a configurable expiry period between 7 and 365 days for each registration link.
5. THE Registration_Service SHALL enforce a configurable maximum-use count per link.
6. WHEN a registration link has reached its maximum-use count, THE Registration_Service SHALL reject further registrations using that link and return a descriptive error.
7. WHEN a Student registers via a link, THE Registration_Service SHALL require only Full Name and Admission Number as input fields.
8. THE Registration_Service SHALL prevent duplicate registrations by rejecting a second registration attempt using the same Admission Number within the same school.
9. WHEN a Teacher adds a Student manually, THE Registration_Service SHALL create the Student record with the Teacher's schoolId, departmentId, and classId without requiring a registration link.

### Requirement 5: QR Code Attendance

**User Story:** As a Teacher, I want to start a QR-based attendance session, so that students can scan a code to mark themselves present without manual intervention.

#### Acceptance Criteria

1. WHEN a Teacher starts an Attendance_Session, THE Session_Service SHALL generate a unique QR_Code linked to that session.
2. THE Session_Service SHALL refresh the QR_Code every 30 seconds to prevent screenshot-based fraud.
3. WHEN a Student scans a valid QR_Code within the active session window, THE Attendance_Service SHALL record the student's attendance with a timestamp.
4. WHEN a Student scans a QR_Code that has expired (older than 30 seconds), THE Attendance_Service SHALL reject the scan and return a descriptive error.
5. WHEN a Student scans a valid QR_Code and the scan time is within the Late_Threshold of the session start time, THE Attendance_Service SHALL record the status as PRESENT.
6. WHEN a Student scans a valid QR_Code and the scan time exceeds the Late_Threshold but is within twice the Late_Threshold, THE Attendance_Service SHALL record the status as LATE.
7. WHEN a Student scans a valid QR_Code and the scan time exceeds twice the Late_Threshold, THE Attendance_Service SHALL record the status as ABSENT.
8. THE Session_Service SHALL validate the student's GPS coordinates at scan time and SHALL reject the scan if the coordinates fall outside a configurable radius from the session location.
9. WHEN a Student scans a QR_Code while offline, THE Client_App SHALL store the scan record in IndexedDB and SHALL sync the record to the server within 30 seconds of the device regaining connectivity.
10. THE Attendance_Service SHALL reject duplicate scans from the same Student within the same Attendance_Session.

### Requirement 6: Manual Attendance Marking

**User Story:** As a Teacher, I want to manually mark attendance for each student in my class, so that I can record attendance when QR scanning is not feasible.

#### Acceptance Criteria

1. WHEN a Teacher opens the manual attendance view for an active session, THE Attendance_Service SHALL return the full student list for that Teacher's assigned class.
2. WHEN a Teacher marks a Student's attendance, THE Attendance_Service SHALL accept one of four statuses: PRESENT, LATE, EXCUSED, or ABSENT.
3. WHEN a Teacher marks a Student as EXCUSED or ABSENT, THE Attendance_Service SHALL accept an optional reason note of up to 500 characters.
4. WHEN a Teacher submits a manual attendance mark while offline, THE Client_App SHALL store the record in IndexedDB and SHALL sync the record to the server within 30 seconds of the device regaining connectivity.
5. WHEN a Teacher updates an existing attendance record within the same session, THE Attendance_Service SHALL overwrite the previous status and log the change in the AuditLog.

### Requirement 7: Biometric Attendance

**User Story:** As a Teacher, I want to use my phone's camera to verify a student's identity biometrically, so that I can mark attendance with high confidence and prevent impersonation.

#### Acceptance Criteria

1. WHERE the school's Plan_Tier is Professional or Enterprise, THE Biometric_Service SHALL be available for attendance marking.
2. WHEN a Teacher initiates a biometric scan, THE Biometric_Service SHALL capture a facial image using FaceAPI running in the browser on the Teacher's device.
3. THE Biometric_Service SHALL perform Liveness_Detection before processing any biometric sample to reject photos, videos, or spoofed inputs.
4. WHEN a biometric sample is captured, THE Biometric_Service SHALL compare it against stored Biometric_Templates for students in the Teacher's class and SHALL return the top match with a confidence score.
5. WHEN the confidence score meets or exceeds the configured minimum threshold, THE Attendance_Service SHALL mark the matched Student as PRESENT.
6. WHEN the confidence score falls below the configured minimum threshold, THE Biometric_Service SHALL reject the match and prompt the Teacher to retry or use an alternative method.
7. WHEN a Teacher performs biometric attendance while offline, THE Client_App SHALL use locally cached Biometric_Templates and SHALL sync the resulting attendance record within 30 seconds of regaining connectivity.
8. THE Biometric_Service SHALL store Biometric_Templates as encrypted data and SHALL never transmit raw biometric samples over the network.

### Requirement 8: Offline-First Operation

**User Story:** As a Teacher or Student, I want the system to work without an internet connection, so that attendance is not disrupted by network outages.

#### Acceptance Criteria

1. WHEN a device loses internet connectivity, THE Client_App SHALL continue to support QR scanning, manual marking, and biometric attendance using locally cached data.
2. WHEN a device regains internet connectivity, THE Sync_Service SHALL automatically upload all pending offline records to the server within 30 seconds.
3. WHEN an offline record and a server record exist for the same Student and Attendance_Session, THE Sync_Service SHALL retain the record with the newer timestamp and SHALL discard the older record.
4. THE Sync_Service SHALL log every Conflict_Resolution decision in the AuditLog with both the offline and server record values.
5. THE Client_App SHALL store all offline data in IndexedDB and SHALL not rely on browser cookies or session storage for offline persistence.

### Requirement 9: Real-Time Attendance Feed

**User Story:** As a Teacher, I want to see attendance updates in real time during an active session, so that I can monitor who has checked in without refreshing the page.

#### Acceptance Criteria

1. WHEN an Attendance_Session is active, THE Realtime_Service SHALL push attendance record updates to all connected Teacher clients subscribed to that session via Socket_IO WebSocket.
2. WHEN a new attendance record is created or updated, THE Realtime_Service SHALL broadcast the update to subscribed clients within 2 seconds.
3. WHEN a Teacher's WebSocket connection is interrupted and then restored, THE Realtime_Service SHALL replay all attendance events that occurred during the disconnection period.
4. WHEN an Attendance_Session is ended by the Teacher, THE Realtime_Service SHALL close the WebSocket subscription for that session and notify all connected clients.

### Requirement 10: Attendance Reports and Export

**User Story:** As a School Admin, I want to generate attendance reports at multiple levels, so that I can monitor attendance trends and identify at-risk students.

#### Acceptance Criteria

1. WHEN a Student requests a personal attendance report, THE Report_Service SHALL return attendance records scoped to that Student's schoolId and studentId only.
2. WHEN a Teacher requests a class report, THE Report_Service SHALL return aggregated attendance data for all students in the Teacher's assigned class.
3. WHEN a HOD requests a department report, THE Report_Service SHALL return aggregated attendance data for all classes within the HOD's assigned department.
4. WHEN a School_Admin requests a school report, THE Report_Service SHALL return aggregated attendance data for all departments within the school.
5. THE Report_Service SHALL calculate each student's attendance percentage using the formula: `(Total Present / Total Expected) × 100`.
6. WHEN a report is exported, THE Report_Service SHALL produce the output in either PDF or Excel format as specified by the requesting user.
7. THE Report_Service SHALL scope all report data to the requesting user's role and schoolId and SHALL not include data from other schools or unauthorized scopes.

### Requirement 11: Dropout Risk Scoring

**User Story:** As a School Admin or HOD, I want the system to calculate a dropout risk score for each student, so that I can intervene early for at-risk students.

#### Acceptance Criteria

1. THE Risk_Service SHALL compute a Dropout_Risk_Score for each Student using the formula: `(Attendance_Weight × 0.4) + (Grade_Weight × 0.4) + (Pattern_Weight × 0.2)`, where each weight is a normalized value between 0 and 100.
2. THE Risk_Service SHALL classify each score into one of four risk levels: Low (0–20%), Medium (21–50%), High (51–80%), or Critical (81–100%).
3. WHEN a School_Admin or HOD requests risk scores, THE Risk_Service SHALL return scores scoped to the requesting user's permitted scope (school or department respectively).
4. THE Risk_Service SHALL recalculate the Dropout_Risk_Score for a Student after each new attendance record is created or updated for that Student.
5. WHEN a Student's risk level changes from one classification to another, THE Notification_Service SHALL send an alert to the Student's assigned Teacher and HOD.

### Requirement 12: Plan Tier Feature Gating

**User Story:** As a Super Admin, I want features to be gated by plan tier, so that schools only access capabilities included in their subscription.

#### Acceptance Criteria

1. THE License_Service SHALL enforce the following student count limits per tier: Trial (50 students, 14 days), Basic (500 students, monthly at KES 5,000), Professional (2,000 students, monthly at KES 15,000), Enterprise (unlimited students, annual at KES 300,000).
2. WHERE the school's Plan_Tier is Trial, THE License_Service SHALL restrict the school's access to QR and GPS attendance methods only and SHALL disable Biometric and AI_Assistant features.
3. WHERE the school's Plan_Tier is Basic or higher, THE License_Service SHALL enable API access for the school.
4. WHERE the school's Plan_Tier is Professional or Enterprise, THE License_Service SHALL enable Biometric attendance and AI_Assistant features.
5. WHERE the school's Plan_Tier is Enterprise, THE License_Service SHALL enable Custom Branding configuration for the school.
6. WHEN a school's student count reaches the plan limit, THE License_Service SHALL prevent new student registrations and SHALL notify the School_Admin with an upgrade prompt.
7. WHEN a school's license expires, THE License_Service SHALL restrict the school's access to read-only mode and SHALL notify the School_Admin with a renewal prompt.

### Requirement 13: Manual Payment Confirmation

**User Story:** As a Super Admin, I want to manually confirm subscription payments, so that schools are activated on the correct plan after payment is verified.

#### Acceptance Criteria

1. WHEN a School_Admin submits a payment request, THE Payment_Service SHALL create a pending Payment record with the school's plan tier, amount, and timestamp.
2. WHEN the Super Admin confirms a payment in the Super_Admin_Panel, THE Payment_Service SHALL update the school's Plan_Tier and license expiry date accordingly.
3. WHEN the Super Admin rejects a payment, THE Payment_Service SHALL leave the school's plan unchanged and SHALL notify the School_Admin of the rejection via email.
4. THE Payment_Service SHALL generate an invoice record for every confirmed payment and SHALL make it available for download by the School_Admin.
5. THE Payment_Service SHALL log every payment event (submission, confirmation, rejection) in the AuditLog with the amount and timestamp.
6. THE Notification_Service SHALL send an email confirmation to the School_Admin's registered email address using Nodemailer when a payment is confirmed.

### Requirement 14: Role-Aware AI Assistant

**User Story:** As a Teacher, I want to ask the AI assistant questions about my class's attendance, so that I can get insights without manually querying reports.

#### Acceptance Criteria

1. WHEN a user submits a query to the AI_Assistant, THE AI_Service SHALL resolve the query using only data within the user's permitted scope as defined by the user's role and schoolId.
2. WHEN a Teacher asks a question about student attendance, THE AI_Service SHALL return results scoped to that Teacher's assigned class only.
3. WHEN a Student asks a question, THE AI_Service SHALL return results scoped to that Student's own records only.
4. THE AI_Service SHALL support the following query types without requiring a paid AI provider: database queries, attendance calculations, comparisons, pattern identification, and report generation.
5. THE AI_Service SHALL accept voice input via the Web_Speech_API and SHALL convert speech to text before processing the query.
6. IF the AI_Service cannot resolve a query within the user's permitted scope, THEN THE AI_Service SHALL return a message stating that the requested data is not available to the user's role.
7. WHERE the school's Plan_Tier is Professional or Enterprise, THE AI_Service SHALL enable predictive analytics including Dropout_Risk_Score trend visualization.

### Requirement 15: Super Admin Control Panel

**User Story:** As the Super Admin, I want a private control panel, so that I can manage all schools, generate licenses, confirm payments, and suspend non-compliant schools.

#### Acceptance Criteria

1. THE Super_Admin_Panel SHALL be accessible only at `super.smart-managment.com` and SHALL not be linked or referenced from `smart-managment.com` or `api.smart-managment.com`.

5. THE Super_Admin_Panel SHALL display aggregated revenue statistics, active school counts, and plan distribution without exposing individual school user data.
6. THE Super_Admin_Panel SHALL provide access to the AuditLog for all schools, filterable by school, date range, and event type.
7. THE Super_Admin_Panel SHALL display a pending payments queue where the Super Admin can confirm or reject payment submissions from School Admins.

### Requirement 16: Audit Logging

**User Story:** As a Super Admin, I want all significant system events to be logged immutably, so that I can investigate incidents and ensure compliance.

#### Acceptance Criteria

1. THE Audit_Service SHALL create an AuditLog entry for each of the following events: user login, user logout, license activation, attendance record creation, attendance record update, payment event, school suspension, role change, and Conflict_Resolution decision.
2. THE Audit_Service SHALL record the following fields in every AuditLog entry: event type, actor userId, actor role, schoolId, timestamp, and a JSON snapshot of the affected resource.
3. THE Audit_Service SHALL not permit deletion or modification of existing AuditLog entries through any API endpoint.
4. WHEN an AuditLog entry is created, THE Audit_Service SHALL assign a monotonically increasing sequence number to ensure ordering.

### Requirement 17: Timetable Management

**User Story:** As a School Admin, I want to create and manage the school timetable, so that attendance sessions are linked to the correct class, subject, and time slot.

#### Acceptance Criteria

1. WHEN a School_Admin creates a timetable entry, THE Timetable_Service SHALL require the following fields: class, subject, teacher, day of week, start time, and end time.

3. WHEN a Teacher starts an Attendance_Session, THE Session_Service SHALL validate that the session corresponds to a timetable entry assigned to that Teacher at the current time.
4. THE Timetable_Service SHALL restrict create, update, and delete operations to School_Admin role only.
5. WHEN a timetable entry is deleted, THE Timetable_Service SHALL preserve all historical Attendance_Sessions linked to that entry.

**User Story:** As a School Admin, I want the system to send automated notifications, so that relevant users are informed of important events without manual follow-up.

### Requirement 19: Security and Data Protection

**User Story:** As a School Admin, I want the system to protect all user data and communications, so that student and institutional data is not exposed to unauthorized parties.

#### Acceptance Criteria

1. THE API SHALL enforce HTTPS for all client-server communication and SHALL redirect plain HTTP requests to HTTPS with a 301 response.
2. THE Auth_Service SHALL hash all passwords using bcrypt with a per-user salt before storing them in the database.
3. THE Biometric_Service SHALL encrypt all Biometric_Templates at rest using AES-256 encryption before storing them in the database.
4. THE API SHALL validate and sanitize all user-supplied input before processing to prevent injection attacks.
5. THE Auth_Service SHALL enforce a maximum of 5 failed login attempts per user account within a 15-minute window, after which THE Auth_Service SHALL lock the account and notify the School_Admin.
6. THE API SHALL apply rate limiting on all public endpoints, rejecting requests that exceed 100 requests per minute per IP address with a 429 Too Many Requests response.
7. THE Data_Layer SHALL not store raw license key values; THE License_Service SHALL store only a hashed reference after activation.

### Requirement 22: Parser and Data Serialization

**User Story:** As a Developer, I want all data exchanged between the frontend and backend to be correctly serialized and deserialized, so that data integrity is maintained across the system.

#### Acceptance Criteria

1. WHEN the API serializes a response object to JSON, THE Serializer SHALL produce a valid JSON string that can be deserialized back to an equivalent object.
2. WHEN the API deserializes a JSON request body, THE Deserializer SHALL produce an object equivalent to the one that was serialized.
3. THE Serializer SHALL format all date and time values as ISO 8601 strings.
4. WHEN the Report_Service generates a PDF, THE PDF_Renderer SHALL produce a valid PDF document that can be opened by standard PDF readers.
5. WHEN the Report_Service generates an Excel file, THE Excel_Renderer SHALL produce a valid `.xlsx` file that can be opened by standard spreadsheet applications.
6. FOR ALL valid attendance record objects, serializing then deserializing SHALL produce an object with identical field values (round-trip property).

# Requirements Document

## Introduction

SAMS (Smart Attendance Management System) is a multi-school, enterprise-grade attendance platform built for educational institutions in Kenya. The system prevents fraudulent attendance through multi-layer validation (QR code, GPS, and biometric), tracks real-time attendance per session, and supports multiple schools under a single platform with complete data isolation. SAMS operates across three subdomains: a public-facing application (sams.ke), a backend API (api.sams.ke), and a private super-admin panel (super.sams.ke). The platform is designed for cloud deployment on a VPS, supports offline-first operation with automatic sync, and integrates role-aware AI assistance.

---

## Glossary

- **SAMS**: Smart Attendance Management System — the platform described in this document.
- **School**: A registered educational institution activated via a license key.
- **License Key**: A one-time-use activation token in the format `XXXX-YYYY-XXXX-XXXX` that encodes school name, plan tier, and expiry date.
- **School Code**: A short, memorable identifier chosen by the School Admin during activation, used by all school users to log in.
- **Super Admin**: The developer-level administrator who manages all schools, generates licenses, and has access to super.sams.ke.
- **School Admin**: The top-level administrator within a single school, responsible for managing HODs, teachers, students, timetables, and school-wide reports.
- **HOD**: Head of Department — manages teachers and students within a specific department.
- **Teacher**: Manages students in an assigned class, starts attendance sessions, and marks attendance.
- **Student**: End-user who scans QR codes and views personal attendance records.
- **Attendance Session**: A time-bounded event started by a Teacher for a specific class and subject, during which attendance is recorded.
- **Registration Link**: A URL generated by a role holder that embeds School ID, Department ID, Class ID, and expiry metadata, used for self-registration.
- **QR Code**: A dynamically refreshing code displayed by the Teacher's device, scanned by students to mark attendance.
- **Biometric Template**: A stored facial or fingerprint data record associated with a Student, used for biometric attendance verification.
- **Liveness Detection**: A biometric check that confirms the subject is a live person, not a photo or recording.
- **Dropout Risk Score**: A computed value (0–100%) indicating a student's risk of dropping out, derived from attendance, grades, and behavioral patterns.
- **IndexedDB**: Browser-based local storage used for offline data persistence on student and teacher devices.
- **Conflict Resolution**: The process of reconciling attendance records that were created both offline and online.
- **M-Pesa**: Safaricom's mobile money platform used for subscription payments.
- **STK Push**: M-Pesa's Sim Toolkit Push — a payment prompt sent directly to a user's phone.
- **Plan Tier**: One of four subscription levels: Trial, Basic, Professional, or Enterprise.
- **Late Threshold**: A configurable time window (in minutes) after session start within which a student is marked PRESENT rather than LATE.
- **Attendance Status**: One of four values: PRESENT, LATE, EXCUSED, or ABSENT.
- **AI Assistant**: The role-aware assistant embedded in SAMS that answers queries based on the user's permission level.
- **Timetable**: A school-wide schedule mapping classes, subjects, teachers, and time slots.
- **AuditLog**: An immutable record of significant system events for compliance and debugging.

## Requirements

---

### Requirement 1: License Key Activation

**User Story:** As a School Admin, I want to activate my school using a license key, so that my institution is registered on SAMS with the correct plan and data isolation.

#### Acceptance Criteria

1. WHEN a user submits a valid license key at `sams.ke/activate`, THE Activation_Service SHALL register the school, create the School Admin account, and mark the license key as used.
2. WHEN a license key is submitted that has already been used, THE Activation_Service SHALL reject the request and return a descriptive error message.
3. WHEN a license key is submitted that is expired, THE Activation_Service SHALL reject the request and return a descriptive error message.
4. WHEN a license key is submitted that is malformed or does not match the expected format `XXXX-YYYY-XXXX-XXXX`, THE Activation_Service SHALL reject the request and return a descriptive error message.
5. THE Activation_Service SHALL never expose the raw license key value in any API response, log entry, or UI element after activation.
6. WHEN activation succeeds, THE Activation_Service SHALL store the School Code chosen by the School Admin and enforce its uniqueness across all schools.
7. WHEN a School Code collision is detected during activation, THE Activation_Service SHALL reject the submission and prompt the School Admin to choose a different code.
8. THE License_Key SHALL encode the school name, plan tier, and expiry date such that THE Activation_Service can decode and validate these values without a database lookup.

---

### Requirement 2: Multi-School Data Isolation

**User Story:** As a Super Admin, I want each school's data to be fully isolated, so that users from one school cannot access or affect data belonging to another school.

#### Acceptance Criteria

1. THE Data_Layer SHALL associate every School, User, Department, Class, Timetable, AttendanceSession, AttendanceRecord, RegistrationLink, and AuditLog record with a `schoolId` field.
2. WHEN any authenticated user makes a data request, THE API SHALL scope all query results to the `schoolId` embedded in the user's JWT token.
3. IF a request attempts to access a resource whose `schoolId` does not match the requesting user's `schoolId`, THEN THE API SHALL return a 403 Forbidden response.
4. THE Super_Admin_Panel SHALL be accessible only via `super.sams.ke` and SHALL require Super Admin credentials that are not associated with any single school.
5. WHEN the Super Admin views school data, THE Super_Admin_Panel SHALL display aggregated or per-school data without merging records across schools.

---

### Requirement 3: Role-Based Access Control

**User Story:** As a School Admin, I want each user role to have clearly defined permissions, so that users can only perform actions appropriate to their role.

#### Acceptance Criteria

1. THE Auth_Service SHALL assign exactly one of the following roles to each user: Super Admin, School Admin, HOD, Teacher, or Student.
2. WHEN a School Admin attempts to manage HODs, Teachers, or Students within the same school, THE API SHALL permit the operation.
3. WHEN a HOD attempts to manage Teachers or Students outside the HOD's assigned department, THE API SHALL return a 403 Forbidden response.
4. WHEN a Teacher attempts to start an Attendance Session for a class not assigned to that Teacher, THE API SHALL return a 403 Forbidden response.
5. WHEN a Student attempts to access another student's attendance records, THE API SHALL return a 403 Forbidden response.
6. THE Auth_Service SHALL embed the user's role, schoolId, departmentId, and classId in the JWT token payload.
7. WHEN a JWT token expires, THE Auth_Service SHALL reject all API requests using that token and return a 401 Unauthorized response.
8. THE Auth_Service SHALL support token refresh using a valid refresh token without requiring the user to re-enter credentials.

---

### Requirement 4: Hierarchical Registration Links

**User Story:** As a Teacher, I want to generate a registration link for my class, so that students can self-register with their school, department, and class pre-filled.

#### Acceptance Criteria

1. WHEN a School Admin generates a registration link, THE Registration_Service SHALL embed the schoolId and set the target role to HOD.
2. WHEN a HOD generates a registration link, THE Registration_Service SHALL embed the schoolId and departmentId and set the target role to Teacher.
3. WHEN a Teacher generates a registration link, THE Registration_Service SHALL embed the schoolId, departmentId, and classId and set the target role to Student.
4. THE Registration_Service SHALL enforce a configurable expiry period between 7 and 365 days for each registration link.
5. THE Registration_Service SHALL enforce a configurable maximum-use count per link, defaulting to the class size plus 10%.
6. WHEN a registration link has reached its maximum-use count, THE Registration_Service SHALL reject further registrations using that link and return a descriptive error.
7. WHEN a Student registers via a link, THE Registration_Service SHALL require only Full Name and Admission Number as input fields.
8. THE Registration_Service SHALL prevent duplicate registrations by rejecting a second registration attempt using the same Admission Number within the same school.
9. WHEN a Teacher adds a Student manually, THE Registration_Service SHALL create the Student record with the Teacher's schoolId, departmentId, and classId without requiring a registration link.

---

### Requirement 5: QR Code Attendance

**User Story:** As a Teacher, I want to start a QR-based attendance session, so that students can scan a code to mark themselves present without manual intervention.

#### Acceptance Criteria

1. WHEN a Teacher starts an Attendance Session, THE Session_Service SHALL generate a unique QR code linked to that session.
2. THE Session_Service SHALL refresh the QR code every 30 seconds to prevent screenshot-based fraud.
3. WHEN a Student scans a valid QR code within the active session window, THE Attendance_Service SHALL record the student's attendance with a timestamp.
4. WHEN a Student scans a QR code that has expired (older than 30 seconds), THE Attendance_Service SHALL reject the scan and return a descriptive error.
5. WHEN a Student scans a valid QR code and the scan time is within the Late Threshold of the session start time, THE Attendance_Service SHALL record the status as PRESENT.
6. WHEN a Student scans a valid QR code and the scan time exceeds the Late Threshold but is within twice the Late Threshold, THE Attendance_Service SHALL record the status as LATE.
7. WHEN a Student scans a valid QR code and the scan time exceeds twice the Late Threshold, THE Attendance_Service SHALL record the status as ABSENT.
8. THE Session_Service SHALL validate the student's GPS coordinates at scan time and SHALL reject the scan if the coordinates fall outside a configurable radius from the session location.
9. WHEN a Student scans a QR code while offline, THE Client_App SHALL store the scan record in IndexedDB and SHALL sync the record to the server within 30 seconds of the device regaining connectivity.
10. THE Attendance_Service SHALL reject duplicate scans from the same Student within the same Attendance Session.

---

### Requirement 6: Manual Attendance Marking

**User Story:** As a Teacher, I want to manually mark attendance for each student in my class, so that I can record attendance when QR scanning is not feasible.

#### Acceptance Criteria

1. WHEN a Teacher opens the manual attendance view for an active session, THE Attendance_Service SHALL return the full student list for that Teacher's assigned class.
2. WHEN a Teacher marks a Student's attendance, THE Attendance_Service SHALL accept one of four statuses: PRESENT, LATE, EXCUSED, or ABSENT.
3. WHEN a Teacher marks a Student as EXCUSED or ABSENT, THE Attendance_Service SHALL accept an optional reason note of up to 500 characters.
4. WHEN a Teacher submits a manual attendance mark while offline, THE Client_App SHALL store the record in IndexedDB and SHALL sync the record to the server within 30 seconds of the device regaining connectivity.
5. WHEN a Teacher updates an existing attendance record within the same session, THE Attendance_Service SHALL overwrite the previous status and log the change in the AuditLog.

---

### Requirement 7: Biometric Attendance

**User Story:** As a Teacher, I want to use my phone's camera to verify a student's identity biometrically, so that I can mark attendance with high confidence and prevent impersonation.

#### Acceptance Criteria

1. WHERE the school's plan tier is Professional or Enterprise, THE Biometric_Service SHALL be available for attendance marking.
2. WHEN a Teacher initiates a biometric scan, THE Biometric_Service SHALL capture a facial image or fingerprint from the Teacher's device.
3. THE Biometric_Service SHALL perform liveness detection before processing any biometric sample to reject photos, videos, or spoofed inputs.
4. WHEN a biometric sample is captured, THE Biometric_Service SHALL compare it against stored Biometric Templates for students in the Teacher's class and SHALL return the top match with a confidence score.
5. WHEN the confidence score meets or exceeds the configured minimum threshold, THE Attendance_Service SHALL mark the matched Student as PRESENT.
6. WHEN the confidence score falls below the configured minimum threshold, THE Biometric_Service SHALL reject the match and prompt the Teacher to retry or use an alternative method.
7. WHEN a Teacher performs biometric attendance while offline, THE Client_App SHALL use locally cached Biometric Templates and SHALL sync the resulting attendance record within 30 seconds of regaining connectivity.
8. THE Biometric_Service SHALL store Biometric Templates as encrypted data and SHALL never transmit raw biometric samples over the network.

---

### Requirement 8: Offline-First Operation

**User Story:** As a Teacher or Student, I want the system to work without an internet connection, so that attendance is not disrupted by network outages.

#### Acceptance Criteria

1. WHEN a device loses internet connectivity, THE Client_App SHALL continue to support QR scanning, manual marking, and biometric attendance using locally cached data.
2. WHEN a device regains internet connectivity, THE Sync_Service SHALL automatically upload all pending offline records to the server within 30 seconds.
3. WHEN an offline record and a server record exist for the same Student and Attendance Session, THE Sync_Service SHALL apply the following conflict resolution rule: if the server record timestamp is newer than the offline record timestamp, THE Sync_Service SHALL retain the server record; otherwise THE Sync_Service SHALL retain the offline record.
4. THE Sync_Service SHALL log every conflict resolution decision in the AuditLog with both the offline and server record values.
5. THE Client_App SHALL store all offline data in IndexedDB and SHALL not rely on browser cookies or session storage for offline persistence.

---

### Requirement 9: Real-Time Attendance Feed

**User Story:** As a Teacher, I want to see attendance updates in real time during an active session, so that I can monitor who has checked in without refreshing the page.

#### Acceptance Criteria

1. WHEN an Attendance Session is active, THE Realtime_Service SHALL push attendance record updates to all connected Teacher clients subscribed to that session via WebSocket.
2. WHEN a new attendance record is created or updated, THE Realtime_Service SHALL broadcast the update to subscribed clients within 2 seconds.
3. WHEN a Teacher's WebSocket connection is interrupted and then restored, THE Realtime_Service SHALL replay all attendance events that occurred during the disconnection period.
4. WHEN an Attendance Session is ended by the Teacher, THE Realtime_Service SHALL close the WebSocket subscription for that session and notify all connected clients.

---

### Requirement 10: Attendance Reports

**User Story:** As a School Admin, I want to generate attendance reports at multiple levels, so that I can monitor attendance trends and identify at-risk students.

#### Acceptance Criteria

1. WHEN a Student requests a personal attendance report, THE Report_Service SHALL return attendance records scoped to that Student's schoolId and studentId only.
2. WHEN a Teacher requests a class report, THE Report_Service SHALL return aggregated attendance data for all students in the Teacher's assigned class.
3. WHEN a HOD requests a department report, THE Report_Service SHALL return aggregated attendance data for all classes within the HOD's assigned department.
4. WHEN a School Admin requests a school report, THE Report_Service SHALL return aggregated attendance data for all departments within the school.
5. THE Report_Service SHALL calculate each student's attendance percentage using the formula: `(Total Present / Total Expected) × 100`.
6. WHEN a report is exported, THE Report_Service SHALL produce the output in either PDF or Excel format as specified by the requesting user.
7. THE Report_Service SHALL scope all report data to the requesting user's role and schoolId, and SHALL not include data from other schools or unauthorized scopes.

### Requirement 11: Dropout Risk Scoring

**User Story:** As a School Admin or HOD, I want the system to calculate a dropout risk score for each student, so that I can intervene early for at-risk students.

#### Acceptance Criteria

1. THE Risk_Service SHALL compute a Dropout Risk Score for each Student using the formula: `(Attendance Weight × 0.4) + (Grade Weight × 0.4) + (Pattern Weight × 0.2)`, where each weight is a normalized value between 0 and 100.
2. THE Risk_Service SHALL classify each score into one of four risk levels: Low (0–20%), Medium (21–50%), High (51–80%), or Critical (81–100%).
3. WHEN a School Admin or HOD requests risk scores, THE Risk_Service SHALL return scores scoped to the requesting user's permitted scope (school or department respectively).
4. THE Risk_Service SHALL recalculate risk scores after each new attendance record is created or updated.
5. WHEN a Student's risk level changes from one classification to another, THE Notification_Service SHALL send an alert to the Student's assigned Teacher and HOD.

---

### Requirement 12: Plan Tier Feature Gating

**User Story:** As a Super Admin, I want features to be gated by plan tier, so that schools only access capabilities included in their subscription.

#### Acceptance Criteria

1. THE License_Service SHALL enforce the following student count limits per tier: Trial (50 students), Basic (500 students), Professional (2,000 students), Enterprise (unlimited).
2. WHERE the school's plan tier is Trial, THE License_Service SHALL restrict the school's access to QR and GPS attendance methods only and SHALL disable Biometric and AI features.
3. WHERE the school's plan tier is Basic or higher, THE License_Service SHALL enable API access for the school.
4. WHERE the school's plan tier is Professional or Enterprise, THE License_Service SHALL enable Biometric attendance and AI Assistant features.
5. WHERE the school's plan tier is Enterprise, THE License_Service SHALL enable Custom Branding configuration for the school.
6. WHEN a school's student count reaches the plan limit, THE License_Service SHALL prevent new student registrations and SHALL notify the School Admin with an upgrade prompt.
7. WHEN a school's license expires, THE License_Service SHALL restrict the school's access to read-only mode and SHALL notify the School Admin with a renewal prompt.

---

### Requirement 13: Payment Processing (M-Pesa)

**User Story:** As a School Admin, I want to pay for my subscription via M-Pesa, so that I can renew or upgrade my plan using mobile money.

#### Acceptance Criteria

1. WHEN a School Admin initiates a subscription payment, THE Payment_Service SHALL trigger an M-Pesa STK Push to the School Admin's registered phone number.
2. WHEN the M-Pesa callback confirms a successful payment, THE Payment_Service SHALL update the school's plan tier and expiry date accordingly.
3. WHEN the M-Pesa callback reports a failed or cancelled payment, THE Payment_Service SHALL leave the school's plan unchanged and SHALL notify the School Admin of the failure.
4. THE Payment_Service SHALL generate an invoice record for every completed payment and SHALL make it available for download by the School Admin.
5. THE Payment_Service SHALL log every payment event (initiation, success, failure) in the AuditLog with the transaction reference, amount, and timestamp.

---

### Requirement 14: Role-Aware AI Assistant

**User Story:** As a Teacher, I want to ask the AI assistant questions about my class's attendance, so that I can get insights without manually querying reports.

#### Acceptance Criteria

1. WHEN a user submits a query to the AI Assistant, THE AI_Service SHALL resolve the query using only data within the user's permitted scope as defined by the user's role and schoolId.
2. WHEN a Teacher asks a question about student attendance, THE AI_Service SHALL return results scoped to that Teacher's assigned class only.
3. WHEN a Student asks a question, THE AI_Service SHALL return results scoped to that Student's own records only.
4. THE AI_Service SHALL support the following query types without requiring a paid AI provider: database queries, attendance calculations, comparisons, pattern identification, and report generation.
5. WHERE the school's plan tier is Professional or Enterprise, THE AI_Service SHALL support natural language conversation using an external language model provider.
6. THE AI_Service SHALL accept voice input via the Web Speech API and SHALL convert speech to text before processing the query.
7. IF the AI_Service cannot resolve a query within the user's permitted scope, THEN THE AI_Service SHALL return a message stating that the requested data is not available to the user's role.

---

### Requirement 15: Super Admin Control Panel

**User Story:** As the Super Admin (Developer), I want a private control panel, so that I can manage all schools, generate licenses, monitor revenue, and suspend non-compliant schools.

#### Acceptance Criteria

1. THE Super_Admin_Panel SHALL be accessible only at `super.sams.ke` and SHALL not be linked or referenced from `sams.ke` or `api.sams.ke`.
2. WHEN the Super Admin generates a new license key, THE License_Generator SHALL encode the school name, plan tier, and expiry date into the key and SHALL store a hashed reference in the database.
3. WHEN the Super Admin suspends a school, THE License_Service SHALL immediately revoke all active sessions for that school and SHALL prevent new logins until the suspension is lifted.
4. WHEN the Super Admin extends a school's license, THE License_Service SHALL update the expiry date and SHALL restore access if the school was in read-only mode due to expiry.
5. THE Super_Admin_Panel SHALL display aggregated revenue statistics, active school counts, and plan distribution without exposing individual school user data.
6. THE Super_Admin_Panel SHALL provide access to the AuditLog for all schools, filterable by school, date range, and event type.

---

### Requirement 16: Audit Logging

**User Story:** As a Super Admin, I want all significant system events to be logged immutably, so that I can investigate incidents and ensure compliance.

#### Acceptance Criteria

1. THE Audit_Service SHALL create an AuditLog entry for each of the following events: user login, user logout, license activation, attendance record creation, attendance record update, payment event, school suspension, and role change.
2. THE Audit_Service SHALL record the following fields in every AuditLog entry: event type, actor userId, actor role, schoolId, timestamp, and a JSON snapshot of the affected resource.
3. THE Audit_Service SHALL not permit deletion or modification of existing AuditLog entries through any API endpoint.
4. WHEN an AuditLog entry is created, THE Audit_Service SHALL assign a monotonically increasing sequence number to ensure ordering.

---

### Requirement 17: Timetable Management

**User Story:** As a School Admin, I want to create and manage the school timetable, so that attendance sessions are linked to the correct class, subject, and time slot.

#### Acceptance Criteria

1. WHEN a School Admin creates a timetable entry, THE Timetable_Service SHALL require the following fields: class, subject, teacher, day of week, start time, and end time.
2. WHEN a timetable entry is created that overlaps with an existing entry for the same teacher, class, or room, THE Timetable_Service SHALL reject the entry and return a descriptive conflict error.
3. WHEN a Teacher starts an Attendance Session, THE Session_Service SHALL validate that the session corresponds to a timetable entry assigned to that Teacher at the current time.
4. THE Timetable_Service SHALL restrict create, update, and delete operations to School Admin role only.
5. WHEN a timetable entry is deleted, THE Timetable_Service SHALL not delete historical Attendance Sessions linked to that entry.

---

### Requirement 18: Notification Service

**User Story:** As a School Admin, I want the system to send automated notifications, so that relevant users are informed of important events without manual follow-up.

#### Acceptance Criteria

1. WHEN a Student's attendance falls below a configurable threshold percentage, THE Notification_Service SHALL send an SMS alert to the Student's registered phone number and an in-app notification to the Student's Teacher and HOD.
2. WHEN a school's license is within 7 days of expiry, THE Notification_Service SHALL send a daily email reminder to the School Admin.
3. WHEN a payment is successfully processed, THE Notification_Service SHALL send an email receipt to the School Admin.
4. THE Notification_Service SHALL use Africa's Talking SMS gateway for all SMS notifications.
5. THE Notification_Service SHALL use Nodemailer for all email notifications.
6. IF the SMS gateway returns a delivery failure, THEN THE Notification_Service SHALL retry the delivery up to 3 times with a 60-second interval between attempts and SHALL log each attempt in the AuditLog.

---

### Requirement 19: Security and Data Protection

**User Story:** As a School Admin, I want the system to protect all user data and communications, so that student and institutional data is not exposed to unauthorized parties.

#### Acceptance Criteria

1. THE API SHALL enforce HTTPS for all client-server communication and SHALL reject plain HTTP requests with a 301 redirect.
2. THE Auth_Service SHALL hash all passwords using a cryptographic hashing algorithm with a per-user salt before storing them in the database.
3. THE Biometric_Service SHALL encrypt all Biometric Templates at rest using AES-256 encryption before storing them in the database.
4. THE API SHALL validate and sanitize all user-supplied input before processing to prevent injection attacks.
5. THE Auth_Service SHALL enforce a maximum of 5 failed login attempts per user account within a 15-minute window, after which THE Auth_Service SHALL lock the account and notify the School Admin.
6. THE API SHALL include rate limiting on all public endpoints, rejecting requests that exceed 100 requests per minute per IP address with a 429 Too Many Requests response.
7. THE Data_Layer SHALL not store raw license key values; THE License_Service SHALL store only a hashed reference after activation.

---

### Requirement 20: Deployment and Infrastructure

**User Story:** As the Developer, I want the system to be deployable on a VPS with automated CI/CD, so that updates can be released reliably without manual server intervention.

#### Acceptance Criteria

1. THE Deployment_Config SHALL support deployment on Ubuntu 22.04 with NGINX as a reverse proxy, PM2 as the process manager, and Certbot for SSL certificate management.
2. THE CI_CD_Pipeline SHALL automatically run tests and deploy to the VPS on a successful push to the main branch via GitHub Actions.
3. THE Deployment_Config SHALL route `sams.ke` to the React frontend, `api.sams.ke` to the Express backend, and `super.sams.ke` to the Super Admin panel.
4. THE Deployment_Config SHALL require a minimum of 2 vCPU, 4 GB RAM, and 20 GB SSD for a functional deployment.
5. WHEN the PM2 process manager detects that a service has crashed, THE Deployment_Config SHALL automatically restart the service and log the crash event.
